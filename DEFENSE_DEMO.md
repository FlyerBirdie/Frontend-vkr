# Демонстрация на защите (краткий сценарий)

## Предусловия

- В `Backend/.env` (из `Backend/.env.example`): `JWT_SECRET_KEY`, `PLANNER_USERNAME` и пароль (`PLANNER_PASSWORD` для локальной разработки или `PLANNER_PASSWORD_HASH` для продакшена), плюс `DATABASE_URL` и `CORS_ORIGINS` под адрес фронта (например Vite `http://localhost:5173`).
- Запущены backend (uvicorn) и фронт `npm run dev` (прокси `/api` → API).
- Выполнен **вход** на странице логина теми же учётными данными, что в `.env` планировщика — иначе защищённые маршруты недоступны.

## Ручной чеклист auth (4 пункта)

1. **Без токена:** в новой вкладке/после очистки `sessionStorage` открыть защищённый маршрут (например заказы) — ожидается перенаправление на `/login`.
2. **Неверный пароль:** отправить форму входа с ошибкой в пароле — 401, сообщение об ошибке, токен не сохраняется.
3. **Успешный вход:** корректные `username`/`password` — переход в приложение; в DevTools → Application → Session Storage есть ключ `planner_access_token` с JWT.
4. **Запросы с Bearer:** во вкладке Network у любого запроса к `/api/...` (кроме login и health) в заголовках есть `Authorization: Bearer <jwt>`; при истечении срока или «битом» токене — 401 и возврат к логину.

## Автотесты и скрипты (без обхода auth)

- Отдельного e2e (Playwright/Cypress и т.п.) в репозитории **нет** — ручная проверка по чеклисту выше.
- `Frontend`: `npm test` (Vitest) — только unit-тесты хелперов и разметки, **без** HTTP к API.
- `Backend`: `pytest` — `tests/test_api_auth.py`: защищённый `/api/orders` без заголовка → 401; с `Authorization: Bearer …` после `POST /api/auth/login` → 200. Остальные тесты домена не ходят по HTTP.

**Сценарий демонстрации функционала** (после входа):

1. В навигации открыть **Рабочие** и **Оборудование** — показать справочники; при необходимости добавить запись кнопкой «Добавить».
2. Открыть **Техпроцессы** → перейти в деталь ТП — показать список задач (последовательность, длительность, профессия, модель оборудования); при необходимости скорректировать или добавить задачу.
3. Открыть **Заказы** — создать новый заказ или отредактировать существующий: прибыль, привязка к ТП, плановое окно заказа (`datetime-local` трактуется как **Europe/Samara (UTC+4)** → в API уходит timezone-aware ISO UTC).
4. Перейти в **Расписание**: задать **плановый период** (тот же пояс Самары) → **Запустить планирование**; дождаться блока «Итог планирования» (период, прибыль, вкл./искл., операции).
5. Переключить вид **Гантт (ресурсы)** — временная диаграмма по выбранной группировке (рабочие / оборудование).
6. Переключить **План по заказам** — те же операции в привязке к заказам.
7. Открыть **Анализ** в шапке (или кнопку «Открыть полный анализ» на странице расписания) — метрики загрузки и отчётные блоки; если в сводке есть **исключённые заказы**, вернуть внимание к таблице «Исключённые заказы» в «Итог планирования» на **Расписание** (частичное включение недопустимо — заказ целиком в расписании или в списке исключённых с причиной).
